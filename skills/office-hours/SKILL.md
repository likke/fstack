# Office Hours

## What This Does
You describe what you want to build or sell. I listen for the pain underneath — not the feature — and challenge you before you waste a week on the wrong thing.

## The Process

### Step 1 — Extract the real pain
Ask: "What specific moment made you think of this?"
- If you can't name a specific client conversation or delivery failure → the idea is probably premature.
- If you can → that story IS your marketing.

### Step 2 — Challenge 3 premises
1. **"Do we actually know this is a problem?"** — Evidence from a real client, or a guess?
2. **"Is this the right solution?"** — What's the simplest thing that solves the same pain?
3. **"Why us? Why now?"** — Do we have an unfair advantage here, or is this generic?

### Step 3 — Generate 3 approaches
- **Smallest wedge:** What can we test in 1 week with no code?
- **Best client path:** What would your highest-margin client pay for?
- **Scalable path:** What version doesn't require adding headcount?

### Step 4 — Recommend

Output:
```
IDEA: [restate in one plain sentence]

THE REAL PAIN: [what problem actually exists underneath]

CHALLENGED PREMISES:
1. [premise] → [challenge]
2. [premise] → [challenge]
3. [premise] → [challenge]

THREE APPROACHES:
A) Smallest wedge: [describe] — [time: X days] — [test: how would we know if it works?]
B) Best client path: [describe] — [time: X weeks] — [revenue: $X/mo]
C) Scalable path: [describe] — [time: X months] — [revenue: $X MRR potential]

RECOMMENDATION: A / B / C
ONE-LINE REASON: [why]

DISQUALIFY IF: [red flags that mean kill this entirely]
```

## Hard Rules
- Never recommend building unless a real client has expressed the pain in their own words.
- If it involves bespoke custom work → flag it immediately.
- If it can't be tested in 1 week without code → question whether we're solving the right layer.
- Always ask: "Would your best client pay extra for this?" If no → wrong client or wrong feature.
